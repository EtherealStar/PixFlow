# Asset Reference Contract

## Purpose

Asset references are the only identifiers that may cross the Web, Conversation, Agent, File, Task, Vision, DAG, and Imagegen boundaries when a user mentions material. A reference identifies a durable business resource; it never contains object-storage keys or file bytes.

## Canonical keys

```text
PACKAGE  package:{packageId}
SKU      package:{packageId}/sku:{urlEncodedSkuId}
IMAGE    package:{packageId}/image:{imageId}
```

- `packageId` and `imageId` are existing stable database identifiers. `skuId` is URL encoded as one path segment.
- `referenceKey` is derived from these identifiers. It is not a new database identity and must not be stored as an independent mutable key.
- The backend is the only producer of serialized `referenceKey` values. Clients store and return them verbatim.
- Parsers reject unknown kinds, missing segments, non-positive numeric identifiers, non-canonical encodings, and resources outside the authenticated administrator's scope.

## Reference view

Every mention candidate and persisted message reference uses:

```json
{
  "referenceKey": "package:123/image:789",
  "kind": "IMAGE",
  "sourceType": "ORIGINAL",
  "displayPath": "summer.zip / SKU-001 / front.jpg"
}
```

`sourceType` is `ORIGINAL` or `GENERATED` for IMAGE references. PACKAGE and SKU always describe the original uploaded-archive namespace.

Messages persist `referenceKey` plus `displayPathSnapshot`. Renames affect new candidates but never rewrite historical message labels. A deleted resource keeps only the minimum identity and display-name tombstone required to render the historical token; no deletion timestamp is retained.

## Expansion semantics

- PACKAGE expands to every processable original image extracted from that archive.
- SKU expands to every processable original image for that SKU in the archive.
- IMAGE resolves to exactly one original or generated image.
- Generated images never enter PACKAGE or SKU expansion, even though they inherit `packageId` and `skuId`.
- Multiple input references are expanded to a union and execution work units are deduplicated by `(packageId, imageId)`.
- Parent and child references may coexist because prompt wording can assign them different intent. Only an exactly equal `referenceKey` is a duplicate.

Expansion and readiness validation happen on the backend. The Web client never enumerates a package to construct an execution request.

## Generated images

A successful generated result becomes a first-class image:

- it receives a new `imageId`; the source image ID is never reused;
- it inherits the source `packageId` and `skuId`;
- it records `sourceTaskId`, `sourceResultId`, and `sourceImageId` as lineage;
- it receives the normal IMAGE reference key and may be mentioned in later conversations;
- it remains browsed under Outputs, grouped by conversation and task, rather than being mixed into the Materials page.

Deleting a source package removes its source archive and original image bytes from Materials. If generated images or historical references remain, File retains only the package name and the minimum original-image identity/name metadata needed for reference namespaces and history rendering. Generated bytes remain available.

Deleting a generated image removes its bytes and processable asset row. Only its reference key and original display name remain for historical message rendering.

## Candidate APIs

The mention picker has two browse roots:

- Materials: package -> SKU -> original image. PACKAGE, SKU, and IMAGE are selectable.
- Outputs: conversation -> task -> generated image. Only IMAGE is selectable.

Empty queries use paginated lazy loading. Non-empty queries perform a backend global search across package names, SKU IDs, original image names, and generated image names. Results are grouped by Materials and Outputs and include full display paths.

Requests accept up to 20 `excludeReferenceKeys`; exactly matching keys are omitted on every page so the user cannot select the same reference twice. Uploading, extracting, failed, cancelled, or deleted resources are not candidates. READY and PARTIAL packages may contribute processable original images.

## Message and tool boundary

Message submission carries prompt text and an ordered `references` array separately. The Agent context renders each display snapshot with its canonical key and tells the model that the same key is accepted by asset tools.

Tools use `referenceKey` for one resource or `referenceKeys` for multiple resources. Asset inspection accepts PACKAGE, SKU, and IMAGE. Deterministic batch processing accepts all three kinds and applies the expansion rules above. A tool that requires one concrete image rejects PACKAGE and SKU rather than guessing.

Imagegen is a special case: expansion may discover many original images, but each proposal and task still owns exactly one source IMAGE and one generated IMAGE. A single Agent turn may expose at most 20 IMAGEGEN proposals. If expansion exceeds 20 images, no IMAGEGEN proposal is published; the Agent reports the count and asks the user to narrow the selection.

## Invariants

1. Display names and object-storage keys are never identities.
2. The Web client never constructs, parses, or expands reference keys.
3. Tools never accept the legacy independent `packageId`, `imageIds`, or `source_image_ids` inputs alongside the new reference contract.
4. Permission and ownership checks run again when a key is resolved for a tool or execution proposal.
5. Deleting task execution history never deletes successful generated images.
